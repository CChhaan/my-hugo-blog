## HTTP 与 HTTPS 的区别

**HTTP**：

- 超文本传输协议，信息以明文传输，存在安全风险。
- TCP 三次握手后即可进行报文传输。
- 端口号为 **80**。

**HTTPS**：

- 在 TCP 与 HTTP 层之间加入 **SSL/TLS 安全协议**，使报文能够 **加密传输**，解决了 HTTP 的安全问题。
- TCP 三次握手后，还需进行 **SSL/TLS 握手过程**，才能进入加密报文传输阶段。
- 端口号为 **443**。
- 需要向 **CA（证书权威机构）** 申请数字证书，以保证服务器身份可信。

## HTTPS 解决的安全问题

HTTP 明文传输存在三种风险：**窃听风险**、**篡改风险**、**冒充风险**。

HTTPS 在 HTTP 与 TCP 层之间加入 SSL/TLS 协议，解决上述风险：

- **信息加密**：交互信息无法被窃取；
- **校验机制**：通信内容无法被篡改；
- **身份证书验证**：防止身份冒充。

对应解决方案：

| 风险类型 | HTTPS 解决方式 | 机制说明       |
| -------- | -------------- | -------------- |
| 窃听风险 | **混合加密**   | 实现信息机密性 |
| 篡改风险 | **摘要算法**   | 确保数据完整性 |
| 冒充风险 | **数字证书**   | 验证身份合法性 |

### 混合加密

通过结合 **对称加密** 与 **非对称加密** 的混合方式，保证信息机密性，防止被窃听。

具体过程：

- 在通信建立前，采用 **非对称加密** 交换「会话秘钥」；
- 在通信过程中，使用 **对称加密** 的「会话秘钥」加密明文数据。

### 摘要算法 + 数字签名

计算内容的唯一「指纹」（哈希值），用于校验数据是否被篡改；

无法通过哈希值推导出原内容。

仅使用哈希算法无法防止中间人替换数据，因此需结合 **非对称加密** 进行身份验证。

非对称加密的密钥类型：

- **公钥**：可公开；
- **私钥**：仅由持有者保管。

加解密方向与目的：

- 公钥加密 → 私钥解密。保证内容安全性
- 私钥加密 → 公钥解密。保证消息来源真实性

私钥用于加密消息的哈希值，公钥用于验证签名的真实性，若能正确解密，说明消息来源可靠。

### 数字证书

**CA（证书认证机构）** 颁发的证书，将服务器公钥封装其中，证明公钥可信。

客户端通过浏览器或操作系统验证证书真实性。只要证书可信，公钥即可信。

## HTTPS 建立连接过程（TLS 握手流程）

总体流程：

- 客户端向服务器索要并验证公钥；
- 双方协商生成「会话秘钥」；
- 使用「会话秘钥」进行加密通信。

握手阶段（四次通信）：

1. ClientHello

客户端发送：支持的 TLS 版本、客户端随机数、支持的密码套件列表。 2. SeverHello

服务器响应：确认 TLS 版本、服务器随机数、确认的密码套件、**服务器数字证书**。 3. 客户端回应

客户端验证证书有效性后，发送：加密的随机数、加密算法变更通知、客户端握手结束通知。

三次随机数用于生成「会话秘钥」。

4. 服务器的最后回应

服务器收到随机数后：计算出「会话秘钥」、发送加密算法变更通知、发送握手结束通知。

至此，TLS 握手结束，之后的通信使用加密的 HTTP 数据。

### 客户端校验证书流程

CA 签发证书过程：

- 打包公钥、用途、颁发者、有效期等信息；
- 计算信息哈希值；
- 用私钥加密哈希值生成签名（Certificate Signature）；
- 将签名附加至证书形成数字证书。

客户端验证过程：

- 客户端计算证书哈希值 **H1**；
- 使用 CA 公钥解密签名，得到哈希值 **H2**；
- 比较 **H1** 与 **H2**：相同 → 证书可信；不同 → 证书不可信。

证书信任链：多数证书由 **中间证书** 签发，而非根证书。

### HTTPS 应用数据的完整性保障

TLS 的两层结构：

- **握手协议**：协商加密算法与生成密钥；
- **记录协议**：加密 HTTP 数据并验证完整性。

记录协议的过程：

- 分割消息为片段并压缩。
- 为每个片段生成 **MAC 值（消息认证码）**。
- 将数据与 MAC 一起通过对称加密。
- 添加报头（类型、版本号、长度）。
- 传递至 TCP 层传输。

## HTTPS 的安全性

中间人攻击过程

- 客户端与伪造的中间人服务器建立 TLS 握手；

- 中间人再与真实服务器建立 TLS 握手；
- 中间人解密客户端数据后再转发，形成“监听通道”。

攻击前提：用户必须 **接受伪造证书**或设备中被植入 **恶意根证书**。

### 抓包工具为何能截取 HTTPS 数据

抓包工具通过 **伪造中间人证书** 实现代理，与中间人攻击原理一致。

实现方式：

- 抓包工具在客户端安装其生成的根证书。

- 客户端信任该根证书
- 工具成为“自签发的 CA”，从而能查看 HTTPS 明文数据。

### 防止中间人抓取数据

安全措施

- 保持计算机安全，防止恶意证书导入；
- 不访问证书非法的网站。

使用 HTTPS 双向认证

- **单向认证**：客户端验证服务端身份；
- **双向认证**：服务端也验证客户端身份；若任一方不可信，通信中止。
