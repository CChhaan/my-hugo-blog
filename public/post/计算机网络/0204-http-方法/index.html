2.4 HTTP 方法

发表于： 2025-09-22 分类于： 计算机网络
字数： 1970 浏览：
本节详细介绍了 HTTP 方法，包括 GET、POST、PUT 和 PATCH 方法，并解释了它们的使用场景和区别。

GET 方法

GET 方法是最常见的 HTTP 方法之一，用于从服务器请求数据，是一种不改变内容的询问信息。

浏览器访问网页时，会向服务器发送 GET 请求，请求网页内容。服务器响应数据（如 HTML、图片或其他文件），浏览器显示结果。

GET 请求不会修改任何数据，仅用于“读取”或检索信息。

请求参数附加在 URL 中，因此可在浏览器地址栏中看到。

常见应用场景：

  • 加载网页：输入 URL 或点击链接时，发起 GET 请求。
  • 从 API 获取数据：开发者常使用 GET 请求从外部服务器获取信息。
  • 搜索查询：搜索引擎查询会通过 GET 请求发送搜索词。
  • 获取文件：用于下载图片、查看 PDF、播放视频等。

使用规范与实践：

  • 仅用于数据检索：避免使用 GET 发送敏感信息（如密码）。
  • 保持 URL 简短清晰：避免数据过长或超出 URL 限制。
  • 启用缓存以提升性能：服务器可设置缓存控制头来优化加载速度。
  • 避免修改数据：GET 是“安全”方法，不应引发数据变更。
  • 谨慎处理敏感数据：GET 请求中的参数可能被记录或保存。

POST 方法

POST 方法用于向服务器发送数据，通常用于表单提交、数据存储或处理。与 GET 不同，POST 请求的数据存储在请求体（body）中，而不是 URL。

GET 与 POST 的对比：

比较项GETPOST
数据位置数据包含在 URL 中数据包含在请求体中
可见性数据在地址栏中可见数据隐藏，不在地址栏中显示
主要用途仅检索数据发送、修改或添加数据
缓存可被缓存不会被缓存
幂等性幂等，多次结果相同非幂等，多次请求结果可能不同

常见应用场景：

  • 提交表单
  • 用户认证（用户名与密码验证）
  • 上传文件
  • 创建新资源
  • 向 API 发送数据
  • 在线购物支付处理

PUT 方法

PUT 方法用于更新或替换服务器上的资源。

它发送的数据告诉服务器“资源应该是什么样子”。

应用场景：

  • 更新资源：用新数据替换现有资源。
  • 创建资源：若资源不存在，则使用数据创建新资源。
  • API 更新操作：常用于更新用户资料、产品详情等。

PUT 与 POST 的对比：

比较项PUTPOST
主要用途更新或替换现有资源创建或提交新资源
数据处理替换整个资源增量添加或修改
幂等性幂等（重复执行结果相同）非幂等（可能创建重复资源）
适用场景更新或完全替换资源创建或提交新条目

PATCH 方法

PATCH 方法用于对资源进行部分更新，而非替换整个资源。

特点：

  • 仅更新特定字段，无需重新发送完整数据。
  • 效率更高，减少带宽消耗。
  • 不会覆盖未包含在请求中的字段。

常见使用场景：

  • 更新特定字段（如更改邮箱、添加标签）。
  • 避免数据丢失（仅修改部分字段）。
  • 适合大型资源的部分更新。

PUT 和 PATCH 之间的关键区别：

特性PUTPATCH
目的替换整个资源部分更新资源
数据处理发送完整资源仅发送需更新的字段
效率对大资源较低对部分更新更高效
幂等性幂等不一定幂等
数据丢失风险可能覆盖字段不会覆盖未指定字段

DELETE 方法

DELETE 方法用于从服务器删除资源，执行后资源将不再可用。

特点：

  • 仅需资源 URL，无需请求体。
  • 操作不可逆，需谨慎使用。

安全处理建议：

  • 需要身份验证：仅授权用户可删除。
  • 使用确认步骤：防止误删。
  • 记录删除操作：便于追踪与恢复。
  • 软删除关键数据：可标记为删除而非彻底移除。
  • 错误处理与反馈：明确返回删除结果。

常见状态码：

  • 200 OK：删除成功，有响应体。
  • 204 No Content：删除成功，无响应体。
  • 404 Not Found：资源不存在。

HEAD 方法

HEAD 方法与 GET 类似，但仅返回响应头信息，不包含内容。

特点：

  • 获取资源元数据（如大小、类型、修改时间）。
  • 更快，节省带宽。
  • 无副作用，安全且幂等。

常见应用场景：

  • 检查资源是否存在。
  • 测试链接有效性。
  • 获取文件元数据。
  • 优化缓存（检查是否更新）。
  • API 状态验证。
  • 服务器健康监控。

OPTIONS 方法

OPTIONS 方法用于查询服务器支持的操作类型，即资源可执行的 HTTP 方法。

工作机制：

  • 客户端发送 OPTIONS 请求询问允许的方法。
  • 服务器以 Allow 头字段返回支持的 HTTP 方法。

应用场景：

  • 发现可用方法。
  • CORS 预检请求：检查跨域访问权限。
  • 提升 API 效率：帮助客户端了解可执行操作。

CORS 相关流程：

  • 浏览器先发送 OPTIONS 预检请求。
  • 服务器返回允许的方法、域名及头信息。
  • 浏览器决定是否继续后续请求。

TRACE 方法

TRACE 方法用于调试 HTTP 请求路径，通过回显服务器收到的请求内容来诊断传输问题。

工作原理：

  • 客户端发送 TRACE 请求。
  • 服务器返回原始请求内容。
  • 可用于检测代理、防火墙或网络修改。

使用价值：请求追踪、网络调试、代理与防火墙分析

安全风险：

  • 可能暴露敏感信息（如 Cookies、Token）。
  • 可被利用进行 XSS 攻击。
  • 多数服务器默认禁用 TRACE。

CONNECT

CONNECT 方法用于建立网络隧道（通常通过代理）以实现安全的加密通信，例如 HTTPS。

工作机制:

  • 客户端发送 CONNECT 请求（包含目标主机和端口）。
  • 代理服务器建立到目标服务器的隧道。
  • 客户端与目标服务器直接进行加密通信（如 TLS）。

典型应用:

  • HTTPS 通过代理
  • VPN 加密通道
  • 访问被封锁内容
  • 自定义代理转发

安全风险与防范：

  • 可能绕过内容过滤系统。
  • 可被用于隧道传输恶意流量。
  • 组织通常监控或限制 CONNECT 的使用。

不同的 HTTP 方法具有明确的用途和边界：

方法主要作用
GET获取数据
POST创建或提交数据
PUT更新或替换资源
PATCH局部更新资源
DELETE删除资源
HEAD获取头部信息
OPTIONS查询支持的方法
TRACE调试请求路径
CONNECT建立安全隧道
