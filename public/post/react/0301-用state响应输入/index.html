3.1 用 state 响应输入

发表于： 2025-10-13 分类于： React文档阅读
字数： 1412 浏览：
本文介绍了 React 中如何用 state 响应输入，包括声明式 UI 与命令式 UI 的比较、声明式地考虑 UI、构建 state 的原则等内容。

声明式 UI 与命令式 UI 的比较

命令式编程：通过明确的步骤告诉计算机“如何”更新 UI，需要直接操作 UI。

声明式编程（React）：只需要声明你想要显示的内容，React 通过计算决定如何更新 UI。

命令式编程的特点：适用于独立系统，在复杂系统中管理难度会指数级增长。

声明式编程的特点：不直接操作 UI，聚焦于描述 UI 的最终状态。

声明式地考虑 UI

步骤 1：定位组件中的不同视图状态

可视化用户可能看到的所有 UI 状态。

模拟可帮助在实现逻辑前快速迭代 UI。

可将多个视图状态展示在同一页面中，如 “living style guide” 或 “storybook”。

步骤 2：确定触发状态改变的原因

触发 state 更新的输入：

  • 人为输入：点击按钮、在表单中输入、导航到链接

    通常需要事件处理函数

  • 计算机输入：网络请求得到反馈、定时器触发、图片加载完成

两种情况都必须通过设置 state 变量来更新 UI。

步骤 3：通过 useState 表示内存中的 state

视图状态需由内存中的 useState 表示。

原则：让“变化的部分”尽可能少，减少潜在 bug。

从“绝对必须”存在的状态开始设计。

使用状态变量表示可视状态，可能存在多种实现方式，需要尝试。

如果无法立即确定最佳结构，先加入足够多的 state，确保覆盖所有可能视图状态。

步骤 4：删除任何不必要的 state 变量

目标：避免 state 中出现重复或无效的内容，确保 state 始终能正确代表 UI。

可自检的问题：

  • 这个 state 是否会导致矛盾？
  • 相同的信息是否已经存在于另一个 state 中？
  • 能否通过另一个 state 的相反值表达同一信息？

原则：只有在无法删除而不破坏功能时，这些 state 才是必要的。

补充机制：可将状态提取到 reducer 中，通过合并多个 state 变量并集中逻辑来实现更精确的模块化。

步骤 5：连接事件处理函数以设置 state

最后，为组件添加事件处理函数以更新 state。

虽然代码可能比命令式方式更长，但更健壮，能避免引入新状态后破坏现有 state，允许修改 UI 而无需变更交互逻辑

构建 state 的原则

当编写存有 state 的组件时，需要决定 state 变量的数量及数据格式。以下原则可帮助做出更好的决策：

  1. 合并关联的 state。

    **当多个 state 变量总是同时更新时，应考虑将它们合并为一个。**这样可以避免忘记保持它们同步。

    当未来可能出现不确定数量的 state 片段时，将数据整合到一个对象或数组会更有帮助（例如用户可添加自定义字段的表单）。

    当 state 是对象时，不能只更新其中一个字段而不显式复制其他字段。

  2. 避免互相矛盾的 state。

    当 state 结构中存在多个矛盾或“不一致”的 state 时，会带来隐患。**应尽量避免互相矛盾的 state。**组件越复杂，越难理解发生了什么。

  3. 避免冗余的 state。

    如果可以在渲染期间根据 props 或现有 state 计算出某些信息，则不应将这些信息存入 state。

    只有在有意忽略某个 prop 的更新时，才应将其“镜像”到 state 中。按惯例，prop 名以 initial 或 default 开头，表示新值将被忽略。

  4. 避免重复的 state。

    如果同一个数据在多个 state 变量或多个嵌套对象中重复存储，会导致难以保持同步。应尽可能减少重复。

  5. 避免深度嵌套的 state。

    深度嵌套的 state 更新不便。如果可能，应该以扁平化方式构建 state。

    理想情况下，删除“表”对象中的已删除项目（及其子项）以改善内存使用。可以使用 Immer 使更新逻辑更简洁。

    通过将一些嵌套 state 移动到子组件中，可以减少嵌套。对于不需要保存的短暂 UI 状态特别有效。

这些原则的目标是使 state 易于更新且不易出错。删除冗余和重复的数据有助于确保各部分保持同步。这类似于数据库工程师对数据库结构进行“规范化”以减少错误出现的机会。“让你的状态尽可能简单，但不要过于简单。”
